# Developer Guide

This document explains the implementation model for developers who need to maintain or extend `webhook-router`.

## Runtime Flow

### Webhook To App Stream

```text
POST /webhooks/{channel_id}
  -> validate Channel and secret
  -> read headers and body
  -> build WebhookEvent with source_id
  -> find enabled routes for channel
  -> write one Redis Stream entry per target App
  -> return 200 OK after Redis writes succeed
```

Webhook responses do not wait for SSE clients or Callback consumers to finish processing. Once the event is accepted into Redis Stream for the routed Apps, the third-party Webhook receives success.

### SSE Delivery

SSE delivery reads directly from Redis Stream using Redis Stream IDs as cursors.

```text
GET /apps/{app_id}/events
  -> validate App and token
  -> if Last-Event-ID exists, replay events after it
  -> XREAD new events from App Stream
  -> emit SSE event: webhook
  -> send heartbeat comments periodically
```

The online connection broker is currently used for connection statistics, not for event delivery. Keeping SSE delivery on Redis Stream avoids split-brain behavior between real-time push and replay.

### HTTP Callback Delivery

Callback delivery is handled by one worker goroutine per App with callback enabled.

```text
worker reads app stream from stored cursor
  -> POST event JSON to callback.url
  -> include X-Relay-Callback-Secret when configured
  -> 2xx means success
  -> non-2xx, timeout, network error retry with exponential backoff
  -> after max_attempts, record permanent failure and advance cursor
```

The callback cursor is stored in Redis key:

```text
{key_prefix}:app:{app_id}:callback_cursor
```

MVP is single-instance. It does not use Redis Consumer Group. If multi-instance deployment is needed later, this worker model must be revisited.

## Event Model

The event struct is defined in `internal/event/event.go`.

```json
{
  "id": "1715330000000-0",
  "source_id": "src_01J00000000000000000000000",
  "channel": "gewe-main",
  "received_at": "2026-05-10T12:00:00Z",
  "headers": {},
  "body": {}
}
```

Field rules:

- `id` is the Redis Stream ID for the App Stream entry. It is filled when reading from Redis.
- `source_id` is generated once per original Webhook. Events routed to multiple Apps share the same `source_id`.
- `channel` is the source Channel ID.
- `headers` are copied from the incoming Webhook request.
- `body` stores parsed JSON or `{ "raw": "..." }` for text payloads.
- `body_base64` is used for non-UTF-8 binary payloads.

## Redis Keys

With default `key_prefix: relay`:

```text
relay:app:{app_id}:events
relay:app:{app_id}:callback_cursor
```

App Stream entries contain one field:

```text
event = <JSON WebhookEvent without id>
```

`id` is intentionally not persisted inside the JSON payload during `XADD`, because the final Redis Stream ID is generated by Redis. The store layer fills `event.ID` from `XMessage.ID` when reading.

## Config Loading

Config loading lives in `internal/config/config.go`.

Responsibilities:

- Load YAML
- Apply safe defaults
- Validate required fields
- Build lookup maps for Channel, App, and Route
- Build routes grouped by Channel

Important behavior:

- `enabled` defaults to true for Channel, App, and Route.
- App must enable at least one delivery mode.
- SSE-enabled App must have `token`.
- Callback-enabled App must have `delivery.callback.url`.
- Duplicate Channel/App/Route IDs fail startup.

## Cleanup Worker

`internal/app/cleanup.go` trims old Redis Stream entries using `cache.ttl`.

It calculates a cutoff timestamp and calls `XTRIM MINID` approximately for each App Stream. This depends on Redis-generated Stream IDs, which encode millisecond time.

## Error Handling Rules

Webhook errors return JSON:

```json
{
  "ok": false,
  "error": "invalid_channel_secret"
}
```

Known error codes include:

- `channel_not_found`
- `channel_disabled`
- `invalid_channel_secret`
- `body_too_large`
- `invalid_body`
- `internal_error`
- `app_not_found`
- `app_disabled`
- `invalid_app_token`
- `sse_not_supported`

## Development Commands

```bash
go test ./...
go vet ./...
go build ./cmd/webhook-router
```

Run with local Redis:

```bash
docker run --rm -p 6379:6379 redis:7-alpine
cp config.example.yaml config.yaml
go run ./cmd/webhook-router --config ./config.yaml
```

## Extension Notes

When adding new features:

- Keep config backward compatible.
- Add default values for new config fields.
- Do not write production secrets into code or docs.
- Preserve existing event fields and semantics.
- If event structure must break compatibility, add `schema_version` first.
- Keep Webhook response independent from consumer business success.
